- 메인 화면 구조는 확정되었다.

- 상세 화면과 일부 이동 규칙은 초안 단계다.

## 화면 설계 원칙

- 첫 화면은 `공장 단위 비교`가 우선이다.
- 개별 센서나 장치보다 `공장 상태`, `이상 시스템`, `변화 흐름`을 먼저 보여준다.
- 내부 Risk Score는 계산되더라도 메인 카드에는 직접 노출하지 않는다.
- `pipeline_status`는 Hub 계산 상태이므로, 현장 입력과 시각적으로 구분되어야 한다.

## 메인 화면 구성

### 상단

- 공장별 위험 상태 카드

표시 항목:
- 공장명
- 현재 상태: `안전 / 주의 / 위험`
- 최근 10분 변화 방향: `상승 / 유지 / 하락`
- 현재 이상 시스템 개수

화면 의도:
- 본사 관제 담당자가 어느 공장을 먼저 볼지 결정한다.

### 중단 왼쪽

- 온도/습도 센서 현황

표시 항목:
- 공장별 현재 온도
- 공장별 현재 습도
- 최근 짧은 추세선

화면 의도:
- 상태 카드에서 본 위험이 환경 요인인지 빠르게 확인한다.

### 중단 오른쪽

- 이상 시스템 목록

표시 항목:
- 공장명
- 시스템 종류
- 현재 상태
- 최근 발생 또는 갱신 시각

정렬 기준:
1. 공장 위험도
2. 같은 위험도 안에서는 최신 발생 순

포함 대상:
- 센서
- 엣지 에이전트
- 노드
- 카메라
- 마이크
- 데이터 수집 파이프라인

### 하단

- 최근 이벤트 또는 상태 변화 로그

표시 항목:
- 공장 위험 상태 변화
- 시스템 정상 -> 이상
- 시스템 이상 -> 복구
- 주요 운영 이벤트

화면 의도:
- 시간 흐름과 상태 전이 맥락을 빠르게 읽는다.

## 상태별 UI 해석

### 공장 카드

- `안전`
  - 현재 즉시 대응 대상이 아님
- `주의`
  - 현재 감시 대상
- `위험`
  - 우선 대응 대상

### 센서 현황

- 값 존재
  - 정상 또는 기준 초과 상태 해석 가능
- 값 `null`
  - 미수신 또는 미사용 필드 가능성
  - TODO: 실제 표기 문구 확정

### 이상 시스템 목록

- 운영형 Spoke(`factory-a`)의 이상은 실제 운영 문제로 해석
- 테스트베드형 Spoke(`factory-b`, `factory-c`)의 이상은 시나리오 또는 검증 문제일 수 있으므로 구분이 필요

### 최근 로그

- 단순 데이터 목록이 아니라, 운영 판단에 필요한 변화만 남긴다.

## 사용자 액션

- 공장 상태를 빠르게 비교
- 이상 시스템 우선순위 식별
- 최근 로그에서 시점 확인

후보 액션:
- 공장 카드 클릭 -> 공장 상세 화면 이동
- 이상 시스템 목록 클릭 -> 시스템 상세 또는 전체 목록 이동
- 로그 클릭 -> TODO: 상세 이동 여부 확인

현재 메모:
- MVP에서는 메인 화면 완성도가 우선이다.
- 상세 이동은 후보로 열어두되, 반드시 구현 대상으로 확정된 것은 아니다.

## 예외 흐름

- 센서 데이터 null
  - `확인 필요` 표시 또는 미수신 상태 표시
- 파이프라인 이상
  - 위험 상태와 별도로 운영 이상 표시

추가 예외:
- Hub가 일부 공장을 인식하지 못함
  - 공장 카드 일부 미노출 또는 상태 미확정
- 테스트베드형 Dummy 전환 중
  - 급격한 상태 변화가 발생할 수 있음
- 전체 데이터 없음
  - 빈 상태 화면 필요
- API 일부 실패
  - 섹션별 부분 실패 허용 여부 확인 필요
